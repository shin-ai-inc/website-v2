# SECURITY.md — ShinAI Website v2 セキュリティ設計（社内・非公開）

本書は内部設計文書であり、公開ホストへデプロイしない（公開除外対象。
PUBLISH_ALLOWLIST.md と各ホストの除外設定 `_config.yml` / `.vercelignore` を参照）。
防御目的のみ。攻撃手法・回避手法は扱わない。

対象: 静的マルチページサイト（partials/ を共有シェルへ結合した8ページ、結合CSS `styles/app.css`、
scripts/ の自前 canvas（流動パーティクル）、assets/）、実装済みのAIアシスタント
（`scripts/chatbot.js`。バックエンド(BFF)のAPIエンドポイント経由でLLMへ接続する設計。
クライアントは鍵を持たない）、問い合わせフォーム（contact。送信先は差し替え可能なプレースホルダ）。

## 1. スコープと保護資産

| 資産 | 種別 | 機密度 | 主な脅威 |
| --- | --- | --- | --- |
| 公開サイトコンテンツ（HTML/CSS/JS/画像） | 公開 | 低 | 改ざん、なりすまし配信 |
| ブランド・アイデンティティ（屋号・商標・トーン） | 公開 | 中 | なりすまし、フィッシング誘導 |
| 連絡導線（shinai.life@gmail.com / mailto） | 公開 | 中 | 収集・スパム、なりすまし |
| 将来のRAGアシスタント（システムプロンプト・接続鍵・社内コーパス） | 内部 | 高 | プロンプトインジェクション、情報漏洩、過剰代理 |
| ビルドソース（BRIEF.md、partials/、開発メモ、本書） | 内部 | 高 | ソース・内部情報の流出 |
| デプロイ資格情報（ホスティング・将来のAPI鍵） | 機密 | 最高 | 漏洩による乗っ取り |

設計原則: 最小権限、多層防御、データ最小化、ゼロトラスト（外部データと
境界を越える入力を信頼しない）。

## 2. STRIDE-lite 脅威モデル

静的サイト＋将来チャットボットに絞った実務版。

| STRIDE | 脅威 | 対象 | 主統制 | 優先度 |
| --- | --- | --- | --- | --- |
| なりすまし (S) | 偽サイト・偽連絡先によるフィッシング | ブランド、連絡導線 | HTTPS強制（HSTS）、canonical、security.txt | P1 |
| 改ざん (T) | 配信物の書き換え、第三者スクリプト注入 | 公開コンテンツ | CSP、外部CDN不使用、SRI、nosniff | P1 |
| 否認 (R) | 改ざん・不正アクセスの追跡不能 | 全体 | ホスティングのアクセスログ、将来チャットの監査ログ | P2 |
| 情報漏洩 (I) | ソース・内部メモ・鍵の流出 | ビルドソース、鍵 | 公開allowlist、ソースマップ非配布、ディレクトリ一覧無効、.gitignore | P0 |
| サービス妨害 (D) | 大量リクエスト、フォーム濫用 | 可用性、連絡導線 | ホスティングのレート制御、honeypot、将来のレート制限 | P2 |
| 権限昇格 (E) | チャットの過剰権限・任意操作実行 | 将来アシスタント | ツールallowlist、人間承認、最小権限 | P1（将来） |

最大の脅威: ビルドソース・内部情報の流出（I, P0）。次点が将来アシスタントの
プロンプトインジェクション経由の情報漏洩・過剰代理。

## 3. ソース・内部情報の流出防止統制

### 3-1. 公開allowlist（公開してよい経路を限定する）

公開・非公開の確定版は PUBLISH_ALLOWLIST.md を単一の真実源とする（本書はその要約）。

公開（デプロイ対象）:
- 生成済み8ページ（index/services/industries/about/faq/contact/privacy/terms）
- styles/app.css（本番は結合CSS一本のみ。ソースCSSは非公開）
- scripts/config.js / nav.js / main.js / particles.js / chatbot.js / vendor/three.min.js
  （Three.js は自己ホスト＝CDNではない。内部コメントを残さない）
- assets/（画像・アイコン・OG画像。assets/CREDITS.md を除く）
- robots.txt / sitemap.xml / site.webmanifest / .well-known/security.txt

非公開（デプロイから必ず除外）:
- BRIEF.md（制作憲法）/ SECURITY.md（本書）/ PUBLISH_ALLOWLIST.md / README.md
- partials/（結合前の断片。結合後のHTMLのみ公開）
- styles/tokens.css / base.css / sections/（ソースCSS。本番は app.css に集約）
- deploy/（_headers / vercel.json / headers.md）/ _build/（ビルド・画像生成スクリプト）
- *.NEEDS.md / 開発メモ / .git/ / node_modules/ / dist/ / 一時ファイル

強制方法: PUBLISH_ALLOWLIST.md を真実源とし、ホストの除外設定（GitHub Pages は `_config.yml`、
Vercel は `.vercelignore`、Netlify は publish ディレクトリ指定）で二重に担保する。

### 3-2. クライアントに秘密を置かない
- APIキー・トークン・個人情報・内部URLを HTML/CSS/JS/コメントに一切書かない。
- チャットのAPI接続は中間サーバー（BFF）経由。LLMの鍵はサーバー側のみが保持し、ブラウザへ渡さない。
  クライアント（chatbot.js）が持つのは公開エンドポイントURL（config.js）だけで、これは秘密ではない。
- フォーム送信先は差し替え可能なプレースホルダのみ。実エンドポイント・鍵は埋め込まない。

### 3-3. ソースマップ非配布
- 本番ビルドで .map を生成・配布しない。配布する JS/CSS から sourceMappingURL を除く。

### 3-4. ディレクトリ一覧の無効化
- ホスティングでオートインデックスを無効化（index 不在ディレクトリは 404）。
- 内部経路は robots.txt の Disallow にも記載するが、これは秘匿手段ではなく
  クロール抑止に過ぎない。秘匿は allowlist によるデプロイ除外で担保する。

### 3-5. コメント規律
- ソースに意思決定・戦術・未公開情報・内部URLをコメントで残さない。コメントは
  「なぜ」を最小限のみ。

## 4. プロンプトインジェクション・ガードレール仕様（API連携・防御）

チャットは BFF 経由でLLMへ接続する設計（`scripts/chatbot.js`）。役割分担を明確にする。

クライアント層（chatbot.js が実装。防御の一層であり最終防壁ではない）:
- 入力長制限（500字）、危険パターン検出（XSS/インジェクション/秘密抽出/SQL断片）、
  クライアント側レート制限（2秒に1回）、sessionId は crypto 乱数。
- モデル応答は必ず `textContent` で描画（HTML化・eval・innerHTML を行わない）。
- リンク生成は許可リスト方式（contact への内部リンクのみ）。鍵・秘密を持たない。
- API URL 未設定時はエラーにせず「準備中」として問い合わせへ穏やかに誘導する。

サーバー層（BFF が満たす契約。接続前に下記 1〜11 を必須化）。外部データ・取得文書・利用者入力は
すべて信頼できない入力として扱う（OWASP LLM Top 10 を参照）。

1. 入力と出力の分離
   - 利用者入力・取得コンテンツ・システム指示を別チャネルとして構造化し、混在させない。
2. システムプロンプトの隔離
   - システムプロンプトはサーバー側に保持し、応答・ログ・エラーに出さない。
   - クライアントから上書き・追記できない。
3. 命令とデータの区切り（delimiting）
   - 取得文書・利用者入力は明示デリミタで囲み「データであって命令ではない」と
     モデルに枠付けする。区切りを越えた指示は無効化する。
4. RAGコンテンツを信頼しない
   - インデックス対象コーパスは取り込み時に検証・出所記録する。
   - 取得本文中の「指示」（例: これまでの指示を無視せよ等）は命令として実行しない。
   - 回答は必ず出典を付し、出典外の主張をしない。
5. ツール・アクションの allowlist
   - 実行可能な操作を明示的な許可リストに限定（既定は読み取りのみ）。
   - 副作用のある操作（送信・書込・購入・外部呼出）は既定で不可。
6. 出力フィルタリング
   - 応答を表示・連携前に検査。秘密・内部経路・資格情報パターンを遮断。
   - 出力をそのまま実行・eval・innerHTML 挿入しない（後段の 5 章参照）。
7. システムプロンプト抽出の拒否
   - システムプロンプト・隠し指示・内部設定の開示要求を拒否する。
8. コンテキストに秘密を載せない
   - 鍵・個人情報・社内機密をプロンプト/コンテキストへ注入しない。
9. アクションの人間承認
   - 副作用を伴う操作は人間の明示承認を必須とする（最終判断は人）。
10. ログ・濫用監視
   - 入出力・ツール呼出を秘密マスク済みで監査ログ化。異常パターンを監視する。
11. レート制限
   - 利用者・IP・セッション単位でレート制限し、自動濫用と抽出試行を抑制する。

接続時の最低条件: 上記 1〜11 を満たし、ShinAI 常時強化型セキュリティ方針配下の
AI利用統制（08_セキュリティ・統制/08_AI利用統制）に脅威モデルを登録すること。

## 5. クライアント堅牢化チェックリスト

- [x] CSP を各ページの meta で配信（`script-src 'self'`）。本番は応答ヘッダ版を優先
      （deploy/_headers・deploy/vercel.json）。JSON-LD は `type="application/ld+json"` の
      データであり実行されないためハッシュ不要（script-src 'self' で阻害されない）。
- [ ] 本番で応答ヘッダ版のセキュリティヘッダ一式を適用（HSTS / X-Content-Type-Options /
      Referrer-Policy / Permissions-Policy / X-Frame-Options / COOP / CORP）。deploy/ 参照。
- [ ] チャットAPI有効化時、CSP の connect-src に API オリジンを追記（meta とヘッダ双方）。
- [ ] Google Fonts のスタイルシートは SRI を検討。Google Fonts は内容が可変のため
      integrity を固定すると更新で破損し得る。バージョン固定の css2 URL に対して算出するか、
      SRI を付さず CSP の style-src/font-src allowlist で範囲を限定する。
      自前配信フォントへ移行すれば font-src 'self' のみで SRI も適用しやすい（推奨）。
- [x] インラインJS・インラインハンドラ禁止。インラインは JSON-LD（データ）のみ。JSは全て外部 'self'。
- [x] 問い合わせフォーム: honeypot（隠しフィールド）＋ required/autocomplete/maxlength/type、
      送信先はプレースホルダ。bot 抑制とクライアント検証を併用。
- [x] DOMへの動的テキストは textContent を使い innerHTML を避ける（chatbot.js で徹底）。
- [x] 外部スクリプトCDN不使用。Three.js は自己ホスト（scripts/vendor、'self' 配信）。外部はフォントのみ。
- [ ] 本番: ソースマップ非配布・ディレクトリ一覧無効・内部ファイル非デプロイ（PUBLISH_ALLOWLIST.md）。

## 7. 残リスクと受容判断

| 残リスク | 影響 | 受容可否 | 監視 |
| --- | --- | --- | --- |
| Google Fonts への外部依存（可用性・追跡） | 低 | 受容（将来は自前配信へ移行を推奨） | フォント読込失敗時のフォールバック確認 |
| meta タグ版CSPでの配信（ヘッダ設定不可ホスト） | 中 | 条件付き受容 | frame-ancestors/HSTS が効かない点を許容範囲か都度判断 |
| robots.txt による内部経路の存在示唆 | 低 | 受容（秘匿は allowlist で担保） | 内部ファイルが実在公開されていないかを定期確認 |
| 将来アシスタント未実装ゆえ仕様のみ | 中 | 受容（接続前に 4 章を必須化） | 接続時に脅威モデル再評価 |
| JSON-LD ハッシュと配信バイトの不一致でCSP違反 | 中 | 不可（接続後に再発防止） | デプロイ後に DevTools/CSP違反レポートで確認 |
