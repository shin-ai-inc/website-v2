# ShinAI Website v2 — 制作ブリーフ（全エージェント共通の憲法）

このファイルはトップページ再構築の単一の真実源。各エージェントは着手前に必ず
`styles/tokens.css` と `styles/base.css` を読み、本ブリーフの規約に従う。

## 0. ポジショニング（迷ったらここへ戻る）
- カテゴリー: 暗黙知AI × 企業専用エージェント
- 企業定義: 企業に蓄積された文書・会話・経験・判断基準を構造化し、検索・判断支援・業務実行に
  使える専用AIへ変換する開発パートナー。
- ブランドPurpose: AIの力で、人間らしい時間を取り戻す。
- 参照モデル: 株式会社スペースデータの「カテゴリー明言 → 仕組みの可視化 → 検証可能な事実で証明」
  という信頼形成の順序を継承する（黒や宇宙の表層は継承しない）。

## 1. 美の基準（全出力に適用）
- 情報の美: 必要なことが必要な順に過不足なく。
- 構造の美: 見た目でなく、読み手が迷わない設計。
- 言葉の美: 上品・誠実・実務的。余計な修飾を削ぐ。
- 余白の美: 詰め込まず呼吸を持たせる。
- 潜在的に、静かな哲学性を一滴。断定や煽りでなく、思想として。

## 2. 改行の作法（重要・採点対象）
- 改行は「文字数の折り返し」ではなく「意味の単位の区切り」。視線移動と認知の流れを設計する。
- 長い文は意味の塊で `<br>` を入れる。ただし入れ過ぎない。媒体（PC/モバイル）で不自然にならぬよう、
  モバイルで崩れる `<br>` は使わず、句で改行したい箇所は別要素（段落・行ブロック）に逃がす。
- 読点は最小限。日本語として自然に。

## 3. 絶対禁止
- 絵文字（置換も不可）。
- 安直な図形マーク（■ □ ● ○ ▲ ★ など）をアイコン代わりに使う。
- AI生成っぽさ（量産アイコン、紫グラデ on 白、Inter/Roboto、定型レイアウト）。
- 胡散臭い断定（「確実なROI」「必ず成果」など）。検証できない数値の断定。
- ダブルクォーテーション " " による強調 → すべて「 」。
- アスタリスク ** ** による強調 → すべて「 」。
- 実績でないものを「導入事例」と呼ぶこと（後述の表記ルール厳守）。

## 4. アイコン/図版/画像の方針（2026-06-27 更新・施主指示反映）
画像の調達順は次の通り。自作の高品位SVGは前提にしない。プロのデザイナーとして
「既にある良い素材を活かし、足りない所だけ補う」方針で品位を担保する。
1. 既存公開サイトの画像を再利用する: `https://shin-ai-inc.github.io/website/assets/images/`
   （確認済み: `logo.png`, `ogp-ai-optimized.jpg`）。ロゴ・OGPはここを正本として `assets/` に取り込む。
2. 新規に画像配置が必要な箇所は、商用利用可・帰属不要の高品質フリー画像で補う
   （Unsplash / Pexels 等のライセンス適合素材）。革新性と信頼性が伝わる、抽象的で上品なものを選ぶ。
   人物の作り物っぽいストック顔写真や、AIくさい定型ビジュアルは避ける。取得元と利用条件を
   `assets/CREDITS.md` に必ず記録する。`assets/` 配下に `webp`/`avif` 等で最適化して置く。
3. アイコンは無理に凝ったSVGを作らない。意味が要る箇所のみ、線画のミニマルな inline SVG を
   軽く用いる（塗りつぶしの量産アイコン・絵文字・■□●○は禁止のまま）。アイコンの作り込み品質に
   時間を使わず、レイアウト・余白・タイポグラフィの完成度を優先する。
- グラデーションは「知識がAIへ変わる」変換点に意味的に限定して使う。装飾は意味のある所だけ。

## 4-2. パーティクル演出の方針（施主が高評価。継承し強化する）
現行ライブサイトは Three.js による 3D 浮遊ジオメトリの「流動的に流れる」パーティクルを持つ。
施主はこの流動感・奥行き・浮遊感を高く評価している。これを「そのまま」継承する（2026-06-27 確定）。
第一原理: 施主が愛する効果を妥協で別物に置換しない。創造性とセキュリティは両立できる。
- 実装は Three.js を「自己ホスト」する（`scripts/vendor/three.min.js`、r134、MIT）。CDNではなく
  自ドメイン配信のため `script-src 'self'` を完全に維持でき、外部依存・第三者追跡もない（CDNより安全）。
- 効果本体は `scripts/particles.js`。二重ワイヤーフレームコア＋多数の浮遊粒子（軌道運動＋波の揺らぎ）＋
  近接接続線＋ポインタ視差＋起動直後の流れる初期モーション。現行の質感を保つ。藍基調に銅を一点。
- 没入型ヒーロー(TOP)の `#three-container` に描画。下層ページは静かな構造を優先しパーティクルを敷かない
  （スペースデータ式: 見せ場はヒーロー、信頼は構造で）。
- prefers-reduced-motion は完全尊重（静止）。タブ非表示で停止。dpr 上限2・端末で粒子数可変で堅牢化。

## 5. タイポグラフィ規約
- 見出し・思想的コピー: `var(--font-display)`（Zen Old Mincho／明朝の品位）。
- 本文・UI: `var(--font-body)`（Zen Kaku Gothic New）。
- 英字ラベル/番号: `var(--font-latin)`（Fraunces）。eyebrow クラスは base.css に既定あり。
- 文字色・サイズ・余白は必ずトークンを使う。直値の color/px を新規に書かない（影/角丸/罫はトークン）。

## 6. 色の規約
- 背景は紙白基調。淡色とグラデを駆使し、単色ベタ塗りを避ける。
- 3ソリューションの色分け（淡く調和）:
  - 暗黙知AI化: `--c-tacit-bg` / `--c-tacit-ink`
  - 企業専用AIエージェント: `--c-agent-bg` / `--c-agent-ink`
  - AI内製化・運用支援: `--c-enable-bg` / `--c-enable-ink`
- 温度の一点に銅 `--copper`。多色化しない。

## 7. マークアップ規約（セクション共通の型）
各セクションは次の骨格に従う。`id` はアンカー兼用。

```html
<section class="section [--alt|--ink]" id="xxx" aria-labelledby="xxx-title">
  <div class="container">
    <div class="section__head reveal">
      <p class="eyebrow">English Label</p>
      <h2 id="xxx-title" class="section__title">日本語見出し</h2>
      <p class="section__lead">必要なら導入文</p>
    </div>
    <!-- 本体。アニメ対象は class="reveal" data-delay="1..5" -->
  </div>
</section>
```

- クラス名はセクション接頭辞で名前空間化（例 `.method__step`, `.solution-card`）。
  グローバルな要素セレクタ（`div`, `p` 単独など）をセクションCSSに書かない（衝突防止）。
- 各セクションCSSは `styles/sections/<name>.css` に1ファイル。トークンのみ参照。
- スクロール表示は `reveal` クラス＋`data-delay`（base.css実装済み）。独自IOは書かない。
- 画像/canvas には必ず代替・aria配慮。装飾は `aria-hidden="true"`。

## 8. アクセシビリティ/レスポンシブ
- モバイル(〜480)、タブレット(〜820)、PC を考慮。レイアウトは grid/flex + clamp。
- ブレークポイントは正準4値のみを使う（中間値を新規に増やさない。tokens.css に定義・根拠）:
  `480`(phone) / `640`(large phone・2列→1列) / `820`(tablet・グローバルナビ切替=JS/footerと一致) / `960`(広グリッド折返し)。
  メディアクエリ条件に CSS 変数は使えないため、各セクションCSSはこの数値で揃える。
- 見出し階層を正しく（h2→h3）。ランドマーク（header/main/footer/nav/section aria-labelledby）。
- コントラスト確保。色だけに依存しない（状態はテキスト/形でも示す）。
- prefers-reduced-motion は base.css で大枠対応済み。重いJS演出は自前でも reduced-motion を尊重。

  - 検証できない定量（○%削減 等）を断定しない。「〜の削減を狙えます」等、目標/可能性として書く。

## 11. ファイル構成（衝突防止のため各エージェントの担当ファイルは排他）
- 共通(オーケストレーター管理): `index.html`, `styles/tokens.css`, `styles/base.css`, `BRIEF.md`
- セクションは `partials/NN-name.html`（`<section>…</section>` 単体）＋ `styles/sections/name.css`。
- JSは `scripts/` 配下に機能単位で。グローバルな scroll-reveal は `scripts/main.js`(オーケストレーター)。

partials は最終的にオーケストレーターが index.html へ結合する。各 partial は head/foot を含めず、
セクション要素のみを書く。CSSは该当セクションのみ。

## 12. セキュリティ規約（全エージェント遵守。防御目的）
攻撃面: 静的マーケサイト＋将来のAIアシスタント(LLM)＋問い合わせフォーム。狙いは
(a) ソースコード・内部情報の流出防止 (b) プロンプトインジェクション耐性 (c) クライアント堅牢化。

### 12-1. 機密・流出防止
- APIキー・トークン・個人情報・内部URLをコード/コメント/ログに一切書かない。
- 公開してよいのは: `index.html`・`styles/`・`scripts/`・`assets/`・法的ページ・`robots.txt`・
  `.well-known/security.txt` のみ。`BRIEF.md`・`partials/`・`SECURITY.md`・開発メモは公開しない
  （デプロイ対象から除外。公開ディレクトリ allowlist をセキュリティ担当が定義）。
- ソースに内部の意思決定・戦術・未公開情報をコメントで残さない。コメントは「なぜ」だけ最小限。
- ディレクトリリスティング無効化・ソースマップ非配布を前提に組む。

### 12-2. CSP と外部依存（インラインJS禁止が前提）
- インライン `<script>` と `onclick` 等のインラインハンドラを書かない。JSは必ず `scripts/` の外部ファイル。
  これにより script-src を 'self' のみに絞れる（'unsafe-inline' を付けない）。
- 外部スクリプトCDNに依存しない。Three.js は CDN から読まず自己ホスト（scripts/vendor/）して 'self' で配信する。
  フォントのみ Google Fonts 許可。
- 想定CSP（セキュリティ担当が確定し、オーケストレーターが head とヘッダ設定へ反映）:
  `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
   font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self';
   base-uri 'self'; form-action 'self'; frame-ancestors 'none'; object-src 'none'`

### 12-3. 入力の取り扱い（フォーム/将来のチャット）
- フォームはクライアント検証＋ハニーポット（隠しフィールド）でbot抑制。`autocomplete`/`maxlength`/
  `pattern` を適切に。送信先は将来差し替え可能なプレースホルダ（実エンドポイント/キーは埋め込まない）。
- DOMに動的テキストを差し込む場合は `textContent` を使い `innerHTML` を避ける（XSS防止）。
- 将来のAIアシスタントは「出典付きで答える」設計とし、プロンプトインジェクション対策を別途仕様化
  （SECURITY.md）。本ビルドでは生のLLM接続を行わない（鍵流出回避）。

### 12-4. 各エージェントの最低遵守
- inline JS/ハンドラ禁止・`innerHTML`回避・秘密を書かない・外部CDNはフォントのみ。
- これらに反する実装はブリーフ違反として却下対象。

## 14. AIチャットボット（現行を継承しバージョンアップ）
現行 `assets/js/chatbot.js`(v3.0.0) を踏襲し、安全性とコード品質を上げて移植する。
- 構成: バックエンドのAPIエンドポイント `${CHATBOT_API_URL}/api/chatbot` に `{message, sessionId}` を
  POST。APIキーはサーバー側のみ。クライアントに鍵を一切埋め込まない（最重要）。
- API URLの設定: インライン禁止のため `scripts/config.js`（外部ファイル）で `window.CHATBOT_API_URL`
  を定義（公開URLであり秘密ではない。実値はデプロイ時に確定。開発中はプレースホルダ）。
- 移植時の改善（技術的負債の解消）:
  - `innerHTML` を全廃し DOM 構築（`textContent`/`createElement`）に統一（XSS面の縮小）。
  - FontAwesome 依存（`<i class="fas">`）を撤廃。ローディングのドットは CSS で描く。
  - CTAリンクは `createElement('a')` で生成。インラインstyleを使わずクラスで。
  - 既存のクライアント側多層防御（長さ制限/危険パターン/レート制限/sessionId は crypto 乱数）を維持・強化。
- UIは紙白テーマに調和（藍基調・銅一点）。`scripts/chatbot.js` と `styles/sections/chatbot.css` で実装。
  HTMLの起動UI（ボタン/ウィンドウ）は共通シェルに含め全ページ共通で出す。

## 15. API連携セキュリティ（プロンプトインジェクション他・防御）
チャットがLLM APIと連携するため、SECURITY.md の脅威モデルに従い多層で守る。要点:
- 鍵はサーバー側のみ。クライアント・リポジトリ・ログに鍵やプロンプト全文を残さない。
- CSP `connect-src` は `'self'` と API オリジンのみ許可（ワイルドカード不可）。
- クライアント入力検証は「防御の一層」であり最終防壁ではない。最終的なプロンプトインジェクション
  対策・出力サニタイズ・レート制限・濫用検知はサーバー側責務（SECURITY.md にサーバー契約を明記）。
- 応答は必ず `textContent` で描画（モデル出力を信用してHTMLにしない）。リンク生成は許可リスト方式。
- これらの詳細・脅威一覧・サーバー側要件は `SECURITY.md` に記す（オーケストレーター/セキュリティ担当）。
