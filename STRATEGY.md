# ShinAI Website — 制作手法の体系化と現状レビュー

本書は BRIEF.md（制作憲法）を補完する**戦略・手法レイヤー**である。
BRIEF が「何を守るか（規約）」を定めるのに対し、本書は「なぜそうするか・どう判断するか
（手法と原理）」を定め、あわせて**現状サイトの実レビュー**を記録する。

- 参照知見: 株式会社スペースデータ（信頼形成の順序）／ Takram（編集的ミニマリズムと情報設計）
- 対象: マルチページ構成・各セクション・情報構造・視覚表現（レスポンシブ）
- 公開境界: 本書は内部判断資料。Web 配信しない（PUBLISH_ALLOWLIST / _config.yml / .vercelignore に登録済み）。

# PART A — 最適手法の体系化

## A-0. 二つの参照モデルから何を継承するか（要点抽出）

我々は表層（黒・宇宙・派手な作家性）を真似しない。**判断の構造**だけを継承する。

### スペースデータ式 — 信頼形成の順序
スタートアップが「実績が薄い段階でも信頼される」ための順序。
1. **カテゴリー明言** — 何の会社かを一文で言い切る。曖昧さを残さない。
2. **仕組みの可視化** — どうやって価値が生まれるかを、段階・図解・プロセスで見せる。
3. **検証可能な事実で証明** — 数値・提携・技術的裏づけ。「言うだけ」にしない。

> ShinAI への翻訳: (1)(2) は既に強い。問題は (3)。確定実績が乏しい（BRIEF 9章）ため、
> **「事実の代替」を技術・プロセス・セキュリティ・人の具体性で埋める**のが本サイト最大の設計課題。

### Takram 式 — 編集的ミニマリズムと情報設計
デザインを「装飾」ではなく「情報の構造そのもの」として扱う流儀。
1. **タイポグラフィ主導** — 文字組と余白で語る。飾りを足さず、削いで品位を出す。
2. **静かな入口・深い中身** — トップは静かな目次。読み応えは下層に置く。
3. **編集としてのケーススタディ** — 事例を「物語（背景→介入→結果）」として構造化する。
4. **バイリンガルの節度** — 英語ラベルは記号として効かせ、和文が主役。

> ShinAI への翻訳: eyebrow の英語ラベル、明朝見出し、白基調・大きな余白は既に Takram 的。
> 不足は (3) の「物語化」。業種別・活用例を before→after の編集体に育てる余地がある。

### 二つの統合 — ShinAI の立ち位置
スペースデータの**論理（信頼の順序）**を骨に、Takram の**佇まい（編集的静けさ）**を肌にする。
派手さではなく「誠実さが美しく見える」状態を目標値に置く。

---

## A-1. 設計の第一原理（迷ったらここへ）

1. **構造が先、装飾は後** — 情報の順序と粒度が決まる前に色や動きを足さない。
2. **証明できないことは約束しない** — 信頼は誇張で買えない。具体と透明性で積む。
3. **余白は機能** — 呼吸は読み手の認知負荷を下げる。詰めるほど安く見える。
4. **一画面に一つの主張** — セクションは「一文で言える主張」を持つ。持てないなら分割か削除。
5. **施主が愛する効果は妥協で置換しない** — パーティクルの流動感は資産。守りながら堅牢化する。

## A-2. 情報構造の手法（IA / セクション設計）

### サイト全体の論理（マルチページ）
各ページに「一つの問い」を割り当て、ページ＝答えの単位にする。

| ページ | 答える問い | 信頼の役割（スペースデータ式） |
| --- | --- | --- |
| index | 何の会社で、なぜ信じられるか | 明言（入口） |
| services | どうやって価値が生まれるか | 仕組みの可視化（中核） |
| industries | 自分の業種で本当に効くのか | 証明の代替（物語化） |
| about | 誰がやっているのか | 証明の代替（人の信頼） |
| faq | 不安・障壁は解消されるか | 障壁除去 |
| contact | どう始められるか | 行動への橋 |

### セクションの黄金骨格
BRIEF 7章の型を、**目的の階層**で運用する。各セクションは次の順に問いに答える。
1. eyebrow（英ラベル）= このセクションの「分類」
2. title（和文見出し）= 主張（一文で言える）
3. lead（任意）= 主張の補足
4. body = 主張の根拠（カード／ステップ／表）

> 判断則: lead が主張の言い換えになっていたら削る。body が主張を支えていなければ並べ替える。

### セクションの「意味の重み」を一定に保つ
スクロールするほど密度が上がる／下がる、の波を作らない。
- 重い（主張＋根拠が多い）セクションの直後は、軽い（一文＋余白）セクションで休符を入れる。
- index の `purpose`（思想・余白）→ `solutions`（情報密度高）の交互は、この休符設計として正しい。

## A-3. 信頼形成の手法（実績が薄い段階の証明）

「証明」の脚が弱いことを正面から設計で解く。**実績の代わりに置けるもの**を順に積む。

1. **プロセスの透明性** — 何を、どの順で、どれくらいの期間で行うか（services の Process）。
   見えないものは怖い。工程を見せること自体が信頼になる。
2. **技術の具体性** — RAG・評価設計・LLM 最適化・オンプレ対応を、固有名詞で語る（capability）。
   「AIで効率化」ではなく「出典を示して答える RAG を構築する」。具体は誇張より強い。
3. **セキュリティの姿勢** — 権限管理・監査ログ・人間承認・データを外に出さない構成。
   B2B の意思決定者が最初に確認する不安に、先回りで答える。
4. **人の信頼** — 代表の経歴（消防士→AI 実務→創業）、CTO の半導体／AI 経験。
   物語のある経歴は、ロゴの羅列より信用される段階がある。
5. **言葉の節度** — 「○%削減」と断定せず「削減を狙えます」。誇張しないことが、逆に信頼を作る。

> 禁則の再確認（BRIEF 9章）: 「導入事例／実績」と断定しない。`stats-bar` のような数値帯は、
> 検証可能な実数が出るまで使わない（base.css に実装はあるが TOP で未使用なのは正しい判断）。

## A-4. 視覚表現の手法（タイポ・色・余白・モーション）

### タイポグラフィ
- 見出し＝明朝（Zen Old Mincho）で品位、本文＝ゴシック（Zen Kaku Gothic New）で可読、
  英ラベル＝Fraunces で抑揚。役割を混ぜない。
- **改行は意味の単位**（BRIEF 2章）。`<br>` ではなく `line` ブロック（`display:block` の span）で
  モバイル崩れを防ぐ現行手法を全ページで徹底する。

### 色（重要な論点 → PART B で指摘）
- 原則は「白基調・淡色とグラデ・単色ベタ塗りを避ける」（BRIEF 6章）。
- **温度の一点**に絞る思想を守る。多色化は安く見える。Takram もスペースデータも色数は少ない。

### 余白
- セクション縦余白は `--section-py`（clamp 5–9rem）で統一。直値の padding を新規に書かない。
- カード内・行間も token 化された space スケールに従い、視覚リズムを一定に保つ。

### モーション
- スクロール表示は `reveal` ＋ `data-delay` の一系統のみ（独自 IO を増やさない）。
- パーティクルは TOP ヒーロー限定。下層は静かな構造を優先（見せ場の一点集中）。
- `prefers-reduced-motion` を全演出で尊重（base.css の大枠＋各 JS 演出で個別に）。

### レスポンシブ
- レイアウトは `grid`/`flex` ＋ `clamp()` を主、メディアクエリは「型が変わる点」だけに使う。
- **ブレークポイントは設計対象**。場当たりに増やさず、少数の意味ある境界に集約する（PART B 指摘）。
- モバイル（〜480）／タブレット（〜820）／PC の三層で必ず実機検証する。

---

## A-5. 制作プロセス（このリポジトリでの運用手順）

1. **着手前に読む**: BRIEF.md → tokens.css → base.css → 本書。規約と原理を頭に入れる。
2. **担当ファイルは排他**: セクションは `partials/<page>.html` ＋ `styles/sections/<name>.css`。
   既存セクション CSS は契約。クラス名を合わせる。足りなければ新規ファイルを所有する。
3. **トークンのみ参照**: color/px/影/角丸/罫を直値で新規に書かない。
4. **ビルドで結合**: `node _build/build.mjs`（styles を app.css へ、partials を共有シェルへ）。
5. **公開境界を確認**: 内部資料・ソース CSS・partials を配信しない（PUBLISH_ALLOWLIST）。
6. **三層で実機確認**: モバイル／タブレット／PC ＋ reduced-motion。

# PART B — 現状サイトのレビュー

対象コミット時点の実装（`partials/`・`styles/`・`scripts/`）を精読した所見。
**良い点は残し、差分と弱点を優先度つきで指摘する**。スコア感: 完成度は高い。残りは「詰め」。

## B-0. 総評
- スペースデータ式の (1) 明言・(2) 可視化は強い。Takram 式の静けさも出ている。
- 弱いのは (3) 証明の脚と、その代替（業種の物語化）。ここが伸びしろ。
- 実装品質は高水準。指摘の多くは「BRIEF と実装のドリフト」と「レスポンシブの整流」。

## B-1. マルチページ構成
**良い点**
- 6 ページ＋法的 2 ページの IA は BRIEF 13章どおり。各ページに問いが割り当たっている。
- 共有シェル（header/footer/chatbot）と partial 契約で、衝突しない分業構造ができている。

**指摘**
- B1-a〔中〕`industries.html` が「Use Cases ＋ Beyond Industry ブリッジ ＋ CTA」の実質2セクション。
  BRIEF 13-1 が求める**業種ごとの before→after の物語**が薄い。証明の脚が最も弱い今、
  ここは A-3 の「物語化」で最も投資価値が高いページ。業種を絞ってでも before→after を厚くする。

## B-2. セクション構造（特に TOP）
**良い点**
- `hero-stack` で「ヒーロー＋信念バンドを一つの量子パーティクル場で包む」構成は美しく、
  施主が愛する流動感を見せ場に集中できている（hero.css）。
- `purpose`（余白・思想）→ `solutions`（密度）の休符設計が効いている（A-2 に合致）。

**指摘**
- B2-a〔高〕**TOP から Problem（3つの痛み）が欠落**。BRIEF 13-1 の TOP 仕様は
  `hero → Problem → Purpose → Solutions → Why → 信頼ストリップ → CTA`。
  実装は `hero → creed → purpose → solutions → capability(Engineering) → why → cta`。
  Problem（退職とともに失われる判断力／文書はあるのに答えが見つからない／入れても使われない）は
  **共感のフック**であり、スペースデータ式でも「課題提示→解決の仕組み」の入口にあたる。
  現状は解決策（solutions）から入るため、読み手が「自分ごと」になる前に機能説明が始まる。
  → 判断: Problem を `purpose` の前後どちらかに一節入れるか、意図的に外したなら BRIEF を更新して
  「実装が正」とする。**どちらにせよ BRIEF と実装の不一致は解消する**。
- B2-b〔中〕TOP の「信頼ストリップ」も未実装。実数がない今は無理に置かない判断は妥当（A-3）。
  ただし BRIEF に残したままだと指示として宙に浮く。BRIEF 側を現実に合わせて整理する。
- B2-c〔低〕`capability`（Engineering）は BRIEF の TOP 仕様には無い追加セクション。
  技術の具体性（A-3-2）を強める良い追加。これは BRIEF に追記して正式化する価値がある。

## B-3. 情報構造・コピー
**良い点**
- 改行が意味単位で `line` ブロック化され、モバイル崩れに強い（BRIEF 2章を全うしている）。
- 言葉の節度（「狙えます」「目指します」）が徹底され、誇張禁則（BRIEF 9章）を守れている。
- eyebrow＝分類／title＝主張／lead＝補足の階層が概ね保たれている（A-2 の骨格に合致）。

**指摘**
- B3-a〔低〕eyebrow の表記揺れ。`contact` のみ `CONTACT`（全大文字）で、他は `Company`/`Solutions`/
  `Industries`/`FAQ` の title case。CSS の `text-transform:uppercase` で見た目は揃うが、ソースの
  一貫性のため title case（`Contact`）に統一する。

## B-4. 視覚的な見た目（色・トークン）
**良い点**
- token 体系が単一真実源として機能し、グラデ・影・角丸まで整理されている（tokens.css）。
- 白基調・淡色・グラデの方針は徹底。AI 生成っぽさを避ける品位がある。

**指摘（重要）**
- B4-a〔高〕**「藍＋銅一点」と実装パレットの乖離**。BRIEF 4-2/6章は「藍基調に銅を一点／
  温度の一点に銅 `--copper`」と規定するが、tokens.css に `--copper` は**存在しない**。
  実装は 藍青(`--brand`) ＋ 青緑(`--teal`) ＋ オレンジ(`--energy`) の三系統＋ソリューション3色分け。
  これは BRIEF が戒める方向（多色化）に寄っている。さらに `.rule-copper` クラスは名前に反して
  銅ではなくブランドグラデ（藍→青緑）を描く（base.css 249行）— 命名の残骸。
  → 判断が必要: (i) 参照2社（少色・静か）に寄せて彩度・色数を絞り BRIEF の「銅一点」に戻すか、
  (ii) 現行の藍青→青緑＋オレンジを正式パレットとして BRIEF を更新するか。
  **施主の美意識（静かな品位 vs 先進的な活気）に関わる方針判断**。実装としては (ii) の方が
  完成しているが、参照モデルの佇まいは (i) 寄り。`.rule-copper` の命名是正は (i)(ii) いずれでも必要。
- B4-b〔低〕`stats-bar` 一式（base.css 211–246）は未使用。実数が出るまで保持は妥当だが、
  当面使わないなら本番 CSS を軽くするため将来削除候補としてメモ。

## B-5. レスポンシブ
**良い点**
- `clamp()` 主導でタイポ・余白が滑らかにスケールし、メディアクエリへの依存が少ない設計思想は良い。
- モバイルナビ（ハンバーガー→全幅展開）、reduced-motion 尊重、`svh`/`lvh` 併用など実装は丁寧。

**指摘**
- B5-a〔中〕**ブレークポイントの断片化**。横断で 480/520/560/600/640/680/720/768/820/900/940 の
  11 種が混在（grep 実測）。意味の境界が散らばり、保守時に「どこで型が変わるか」が読めない。
  → BRIEF 8章の三層（〜480／〜820／PC）に合わせ、token 化した少数境界（例: 480・768・1024）へ
  集約する。各セクションは原則この境界のみを使う。
- B5-b〔中〕**モバイルナビの top 値ハードコード**。header.css 149行 `inset: 78px 0 auto 0` は
  `--header-h`(78px) と一致するが、スクロールで condense すると header は 64px（同 38行）に縮む。
  縮んだ状態でメニューを開くと、ナビ上端と header 下端に約 14px のズレ／隙間が生じうる。
  → `inset: var(--header-h) ...` で変数化し、condense 時も実高に追従させる（JS で高さを CSS 変数に
  反映するか、condense 用の値を別途用意する）。
- B5-c〔低〕ヒーロー見出しは `min-width:520px` で `white-space:nowrap`（hero.css 87行）。
  現行コピーは短く問題ないが、将来見出しを長くすると 520–820px 帯で溢れる可能性。
  コピー変更時は実機（特に 600–768px）で折返しを確認するルールを残す。

## B-6. アクセシビリティ・セキュリティ（横断・簡評）
- ランドマーク、`aria-labelledby`、skip-link、focus-visible、装飾の `aria-hidden` は概ね適切。
- インライン JS 不使用・Three.js 自己ホスト・鍵をクライアントに置かない方針は実装と一致。良好。
- 継続確認: 色のみに依存しないこと（状態をテキスト/形でも示す）と、画像の代替の網羅。

# PART C — 優先アクション（提案）

着手順。各項目は BRIEF と実装の整合を取りつつ、参照2社の原理に寄せる方向。

1. 〔高〕**TOP の Problem 欠落を解消**（B2-a）— 共感の入口を戻すか、外す決定を BRIEF に反映。
2. 〔高〕**色方針の確定**（B4-a）— 「藍＋銅一点」か「藍青→青緑＋オレンジ」か。施主判断を仰ぐ。
   決定後、`--copper` 復活 or `.rule-copper` 改名で命名と実装を一致させる。
3. 〔中〕**industries の物語化**（B1-a）— 業種を絞り before→after を厚く。証明の脚を補強。
4. 〔中〕**ブレークポイント集約**（B5-a）と**モバイルナビ top 変数化**（B5-b）。
5. 〔中〕**BRIEF と実装の差分を整理**（B2-b/B2-c）— capability を正式化、信頼ストリップを現実に合わせる。
6. 〔低〕eyebrow 表記統一（B3-a）、`stats-bar` の去就（B4-b）、ヒーロー nowrap の注意書き（B5-c）。

> 高優先の 1・2 は**方針判断**を含むため、実装前に施主の意向を確認するのが望ましい。
> 3 以降は判断が固まっているため、合意後すぐ着手できる。
